  class Vektor:

   self.x = x
   self.y = y

   return f"Vektor({self.x}, {self.y})"
   def __add__(self, other): # ermöglicht v1 + v2
   return Vektor(self.x + other.x, self.y + other.y)
   def __len__(self): # ermöglicht len(v)
   return 2
   def __eq__(self, other): # ermöglicht v1 == v2
   return self.x == other.x and self.y == other.y
  v1 = Vektor(1, 2)
  v2 = Vektor(3, 4)
  print(v1) # Vektor(1, 2) – ruft __str__ auf
  print(v1 + v2) # Vektor(4, 6) – ruft __add__ auf
  print(len(v1)) # 2 – ruft __len__ auf
  print(v1 == v1) # True – ruft __eq__ auf

  Vorschlag Modul 5: Pythonische Entwurfsmuster

  The lecture notes explicitly require Singleton, Factory, Observer. With 7
  methods (~30 lines, ~4 minutes each) to fill, the question is what to add
  around those three. The "roter Faden" of the course — "Was würde ich in Java
  tun – und wie denkt Python das?" — works best when each pattern has both a
  Java-style implementation AND a Pythonic alternative.

  Empfohlener Aufbau (7 Abschnitte, ~250 Zeilen, ~30 Min)

  1. Singleton — Java-Stil mit __new__ → Python-Modul als Singleton →
  Decorator-Variante
  2. Factory — Java-Stil Klassenhierarchie → Python Dict-Dispatch / Registry
  3. Observer — Java-Stil mit Listener-Interface → Python mit Callable-Liste
  4. Strategy — Java-Stil Strategy-Interface → Python: Funktionen als Objekte
  5. Decorator (Pattern) — GoF-Decorator als Klasse → Python @decorator-Syntax
  6. Iterator — Java-Stil __iter__/__next__ → Generator mit yield
  7. Context Manager — Python-spezifisch, kein direktes Java-Pendant
  (with/__enter__/__exit__)

  Trade-offs, die ich abstimmen möchte

  Trade-off A — Breite vs. Tiefe bei den "Big 3":
  Variante A1 (oben): Big 3 + 4 weitere Patterns → mehr Breite, weniger
  Singleton-Varianten
  Variante A2: Singleton/Factory/Observer mit je 2–3 Python-Varianten, dafür nur
   1–2 weitere Patterns

  Trade-off B — Iterator-Abschnitt:
  Generatoren wurden in Modul 4 schon kurz angeschnitten (Duck Typing, Dunder).
  Falls dort __iter__ bereits behandelt wurde, könnten wir Iterator ersetzen
  durch z.B. Template Method oder Command. Ich kenne deinen Modul-4-Stand nicht
  — wenn du das weißt, hilft das.

  Trade-off C — Context Manager:
  Stark als pädagogischer Moment ("Python hat etwas, das Java so nicht kennt"),
  aber streng genommen kein klassisches GoF-Pattern. Falls dein Prof. nur
  "echte" Patterns will, würde ich Context Manager rauslassen und stattdessen
  Adapter oder Template Method nehmen.
